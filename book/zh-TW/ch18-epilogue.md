# 第十八章：我們學到了什麼

## 五個架構賭注

Claude Code 並非唯一的代理式系統，也不是第一個。但它做出了五個架構賭注，使其有別於眾多 agent 框架。經歷了近兩千個檔案與十七個章節之後，這些賭注值得仔細審視。

### 賭注一：生成器迴圈優於回呼

大多數 agent 框架提供的是一條管線：定義工具、註冊處理器、讓框架進行編排。開發者撰寫回呼函式，框架決定何時呼叫它們。

Claude Code 反其道而行。`query()` 函式是一個非同步生成器——開發者擁有迴圈的控制權。模型串流回應、生成器 yield 工具呼叫、呼叫端執行它們、將結果附加到對話中，然後生成器繼續迴圈。只有一個函式、一條資料流、一個所有互動都必經的地方。生成器回傳型別中的 10 種終止狀態與 7 種延續狀態，編碼了所有可能的結果。迴圈就是系統本身。

這個賭注是：一個單一的生成器函式——即使膨脹到 1,700 行——也會比分散式的回呼圖更容易理解。在研究過原始碼之後，這個賭注確實得到了回報。當你想要理解一個工作階段為何結束，你只需查看一個函式。當你想要新增一種終止狀態，你只需在一個判別聯合型別中新增一個變體。型別系統強制執行窮舉處理。回呼架構會將這些邏輯分散到數十個檔案中，而回呼之間的交互關係將是隱式的，無法在控制流中直接看見。

### 賭注二：基於檔案的記憶優於資料庫

第十一章已詳細闡述了這一點，但其架構意義遠超記憶系統本身。選擇使用純 Markdown 檔案而非 SQLite、向量資料庫或雲端服務，是一個以透明性換取功能性的賭注。資料庫能支援更豐富的查詢、更快的查找與交易保證。檔案則不提供這些。檔案提供的是信任。

一個使用者在 vim 中開啟 `~/.claude/projects/myapp/memory/MEMORY.md`，看到代理確切記住了關於他們的什麼內容——這與一個必須問代理「你記得什麼？」然後祈禱答案是完整的使用者，有著根本性的不同關係。基於檔案的設計使代理的知識狀態可從外部觀察，而非僅僅是自我報告。這比查詢效能重要得多。LLM 驅動的召回系統以檢索智慧彌補了儲存的簡單性——一個 Sonnet 側查詢從清單中選出五筆相關記憶，比嵌入向量相似度更為精確，且不需要任何基礎設施。

### 賭注三：自我描述的工具優於中央編排器

Agent 框架通常提供一個工具註冊表：你在中央配置中描述你的工具，框架再將它們呈現給模型。Claude Code 的工具則是自我描述的。每個 `Tool` 物件攜帶自己的名稱、描述、輸入 schema、提示詞貢獻、並行安全旗標與執行邏輯。工具系統的職責不是向模型描述工具——而是讓工具描述自己。

這個賭注在可擴展性上得到了回報。MCP 工具（第十五章）成為一等公民，因為它們實作了相同的介面。來自 MCP 伺服器的工具與內建工具在模型眼中毫無分別。系統不需要一個獨立的「MCP 工具適配器」層——包裝產生一個標準的 `Tool` 物件，從那一刻起，現有的工具管線就能處理它：權限檢查、並行執行、結果預算控制、Hook 攔截。

### 賭注四：分支代理以共享快取

第九章涵蓋了分支機制：一個子代理以父代理完整的對話作為其上下文視窗的起點，共享父代理的提示詞快取。這不是一個便利性的優化——它是一個架構賭注，賭的是快取共享模型值得承受分支生命週期管理的複雜性。

替代方案——產生一個全新的代理，附帶對話摘要——更簡單但更昂貴。每個全新代理都要從頭付出處理上下文的全部成本。分支代理則免費取得父代理的快取前綴（輸入 token 享有九折優惠），使得為小型任務產生代理變得經濟實惠：記憶萃取、程式碼審查、驗證流程。背景記憶萃取代理（第十一章）在每個查詢迴圈輪次後執行，其成本微乎其微，正是因為它共享了父代理的快取。若沒有基於分支的快取共享，該代理的成本將高到令人卻步。

### 賭注五：Hook 優於外掛

大多數可擴展性系統使用外掛——在宿主程序內部註冊能力並執行的程式碼。Claude Code 使用 Hook——在生命週期節點執行的外部程序，透過結束碼與 stdin/stdout 上的 JSON 進行通訊。

這個賭注是：程序隔離值得付出額外開銷。外掛可能使宿主崩潰。Hook 只會使自己的程序崩潰。外掛可能將記憶體洩漏到宿主的堆積中。Hook 的記憶體隨其程序結束而消亡。外掛需要一個必須版本控制與維護的 API 介面。Hook 只需要 stdin、stdout 與結束碼——一個自 1971 年以來就穩定不變的協定。

開銷是真實的：每次 Hook 調用產生一個程序，需要數毫秒的成本，而程序內回呼則不需要。內部回呼的 -70% 快速路徑（第十二章）顯示系統深知這項成本的重要性。但對於外部 Hook——使用者腳本、團隊 linter、企業策略伺服器——隔離保證使得系統擴展更加安全。企業可以部署基於 Hook 的策略強制執行，而不必擔心格式錯誤的 Hook 腳本會使開發者的工作階段崩潰。

## 哪些可遷移，哪些不可

並非 Claude Code 中的每個模式都具有普適性。有些是規模、資源或特定約束下的產物，其他 agent 建構者未必會面臨相同的條件。

### 可遷移至任何代理的模式

**生成器迴圈模式。** 任何需要串流回應、處理工具呼叫、管理多種終止狀態的代理，都能從將迴圈顯式化而非隱藏在回呼之後中獲益。判別聯合回傳型別——精確編碼迴圈為何停止——是一種能消除整個類別的「代理為何停止了？」除錯工作的模式。

**基於檔案的記憶搭配 LLM 召回。** 具體的實作細節屬於 Claude Code，但原則——簡單的儲存結合智慧的檢索——適用於任何需要跨工作階段持久化知識的代理。四種類型的分類法（使用者、回饋、專案、參考）與可推導性測試（「這是否能從目前專案狀態重新推導？」）是可重複使用的設計啟發法。

**遠端執行的非對稱讀寫通道。** 當讀取是高頻率串流而寫入是低頻率 RPC 時，將兩者分離無論使用何種具體傳輸協定都是正確的。

**位元圖預過濾搜尋。** 任何在大型檔案索引中進行搜尋的代理，都能受益於 26 位元字母位元圖作為預過濾器。每個條目四個位元組、每個候選項一次整數比較——成本效益比令人驚嘆。

**將提示詞快取穩定性視為架構考量。** 如果你的代理使用具有提示詞快取功能的 API，將穩定內容置前、易變內容置後來構建提示詞，這不是一項優化——這是一個決定你成本結構的架構決策。

### 專屬於 Claude Code 規模的模式

**分支的終端渲染器。** Claude Code 分支了 Ink 並以緊湊型別化陣列、基於池的字串駐留與儲存格級差異比對重新實作了渲染管線，因為它需要在終端中達到每秒 60 幀的串流效果。大多數代理渲染到 Web 介面或簡單的日誌輸出。這項工程投資僅在終端渲染是主要 UI 且以高頻率串流時才有意義。

**50 多個啟動效能分析檢查點。** 當你有數十萬使用者且 0.5% 的取樣能產生統計顯著的數據時才有意義。對於較小規模的代理，更簡單的計時系統就已足夠。

**八種 MCP 傳輸類型。** Claude Code 支援 stdio、SSE、HTTP、WebSocket、SDK、兩種 IDE 變體與一個 Claude.ai 代理，因為它必須整合每一種部署拓撲。大多數代理只需要 stdio 與 HTTP。

**Hook 快照安全模型。** 在啟動時凍結 Hook 配置且永不隱式重新讀取，是針對特定威脅的防禦：惡意儲存庫程式碼在使用者接受信任對話框之後修改 Hook。這在你的代理執行於包含不受信任的 `.claude/` 配置的任意儲存庫中時才重要。僅在受信任環境中執行的代理可以使用更簡單的 Hook 管理。

## 複雜性的代價

近兩千個檔案。這換來了什麼，又付出了什麼代價？

檔案數量作為複雜性指標是具有誤導性的。其中大部分是測試基礎設施、型別定義、配置 schema 與分支的 Ink 渲染器。實際的行為複雜性集中在少數幾個高密度檔案中：`query.ts`（1,700 行，代理迴圈）、`hooks.ts`（4,900 行，生命週期攔截系統）、`REPL.tsx`（5,000 行，互動式編排器），以及記憶系統的提示詞建構函式。

複雜性來自三個來源，各有不同的性質：

**協定多樣性。** 支援五種終端鍵盤協定、八種 MCP 傳輸類型、四種遠端執行拓撲與七種配置範疇，本質上就是複雜的。每增加一個協定是對程式碼庫的線性增長，而非指數增長——但總量是龐大的。這種複雜性在 Brooks 的意義上是偶然的：它來自環境（終端碎片化、MCP 傳輸演進、遠端部署拓撲），而非來自所解決的問題本身。

**效能優化。** 基於池的渲染、位元圖搜尋預過濾、黏性快取閂鎖與推測性工具執行，每一項都以複雜性換取可測量的效能提升。這種複雜性經由測量得到了正當化——每項優化之前都有效能分析數據識別出了瓶頸。風險在於，優化會累積並以使熱路徑更難修改的方式相互作用。

**行為調校。** 記憶系統的提示詞指令、過期警告、驗證協定、「忽略記憶」反模式指令——這些不是程式碼複雜性。它們是提示詞複雜性，承載著不同的維護負擔。當模型的行為在版本之間發生變化時，經過評估精心調校的提示詞指令可能需要重新調校。評估基礎設施（在整個程式碼庫中以案例編號與評估分數的形式被引用）是防止退化的防線，但它需要持續的投入。

這個系統的維護負擔是相當可觀的。一位新工程師在閱讀程式碼庫時，必須理解的不僅是程式碼路徑，還有激發特定提示詞措辭的評估結果、激發特定安全檢查的生產事故，以及激發特定優化的效能分析檔案。程式碼註解是詳盡的——許多包含了評估案例編號與前後對比測量——但近兩千個檔案中的詳盡註解本身就是一種閱讀負擔。

## 代理式系統的未來走向

從 Claude Code 中的模式可以觀察到四個趨勢，它們指向了這個領域的發展方向。

### MCP 作為通用協定

第十五章描述了 Claude Code 作為最完整的 MCP 客戶端之一。其重要性不在於 Claude Code 的實作——而在於 MCP 的存在本身。一個標準化的工具發現與調用協定意味著，為一個代理建構的工具可以用於任何代理。生態系統效應是顯而易見的：一個為 Postgres 建構的 MCP 伺服器，一旦完成，就能服務每一個使用 MCP 的代理。開發者在工具整合上的投資是可移植的。

對代理建構者的啟示：如果你正在定義一個自訂的工具協定，你很可能正在犯錯。MCP 已經夠好了，它正在變得更好，而標準協定的生態系統優勢會隨時間複利增長。建構一個 MCP 客戶端，貢獻於規格，讓協定透過社群回饋來演進。

### 多代理協調

Claude Code 的子代理系統（第八章）、任務協調（第十章）與分支機制（第九章）是多代理模式的早期實作。它們解決了特定問題——快取共享、平行探索、結構化驗證——但也揭示了根本挑戰：協調開銷。

代理之間的每則訊息都消耗 token。每次分支共享快取但增加了一個對話分支，父代理最終必須協調整合。Task 系統的狀態機（queued、running、completed、failed、cancelled）是增加複雜性而未增加能力的協調機制。隨著代理變得更加強大，壓力將從「我們如何協調多個代理？」轉向「我們如何使單一代理足夠強大，以至於不需要協調？」

目前的證據表明，兩種方法將共存。簡單任務將使用單一代理。複雜任務將使用協調的多代理系統。工程挑戰在於使協調開銷足夠低，使得交叉點有利於多代理處理真正的平行工作，而不僅僅是處理複雜的任務。

### 持久化記憶

Claude Code 的記憶系統是持久化代理記憶的第一版。基於檔案的設計、四種類型的分類法、LLM 驅動的召回、過期系統，以及用於長時間執行工作階段的 KAIROS 模式，都是一個將持續顯著演進的問題的第一代解決方案。

未來的記憶系統可能會加入結構化檢索（目前系統檢索整個檔案；未來系統可能檢索特定事實）、跨專案遷移學習（適用於所有地方的使用者偏好、不適用於其他專案的專案慣例），以及協作式記憶（第十一章的團隊記憶是第一步，但同步、衝突解決與存取控制都是最小化的）。

開放的問題是基於檔案的方法是否能規模化。在每個專案 200 筆記憶時，它運作良好。在每個專案 2,000 筆記憶時，Sonnet 側查詢的清單變得過大、合併變得過於昂貴、索引超出其上限。基於檔案而非資料庫的架構賭注將在使用量增長時面臨最嚴峻的考驗。

### 自主運作

KAIROS 模式、背景記憶萃取代理、自動 dream 合併、推測性工具執行——這些都是邁向自主運作的步驟。代理在未被要求的情況下做有用的工作：它記住了你忘記告訴它去記住的事情，它在你入睡時合併自己的知識，它在當前回應完成之前就開始執行下一個工具。

軌跡是清晰的。未來的代理將更少被動、更加主動。它們將注意到使用者未曾描述的模式，建議使用者未曾要求的修正，並在無需明確 `/remember` 指令的情況下維護自己的知識。Claude Code 的記憶系統，連同其背景萃取安全網與提示詞工程化的「什麼該保存」啟發法，就是這個未來的原型。

約束在於信任。自主運作要求使用者信任代理在無人值守時會做正確的事。基於檔案的記憶、可觀察的 Hook 系統、過期警告、權限對話框——所有這些的存在都是因為信任必須被贏得，而非被假定。通往更自主代理的道路，必經更透明的代理。

## 結語

十七個章節。六個核心抽象。一個居於中心的生成器迴圈，工具向外延伸，記憶穿越時間回溯，Hook 守衛著邊界，渲染引擎將這一切轉譯為螢幕上的字元，而 MCP 將其連接到程式碼庫之外的世界。

Claude Code 中最深層的模式不是任何單一技術。它是一個反覆出現的決策：將複雜性推向邊界。渲染系統將複雜性推向池與差異比對——管線內部一切都是整數比較。輸入系統將複雜性推向分詞器與按鍵綁定解析器——處理器內部一切都是型別化的動作。記憶系統將複雜性推向寫入協定與召回選擇器——對話內部一切都是上下文。代理迴圈將複雜性推向終止狀態與工具系統——迴圈內部，僅僅是：串流、收集、執行、附加、重複。

每一道邊界吸收混沌，輸出秩序。原始位元組變成 `ParsedKey`。Markdown 檔案變成被召回的記憶。MCP JSON-RPC 變成 `Tool` 物件。Hook 結束碼變成權限決策。在每道邊界的一側，世界是混亂的——五種鍵盤協定、脆弱的 OAuth 伺服器、過期的記憶、不受信任的儲存庫 Hook。在另一側，世界是型別化的、有界的、並被窮舉處理的。

如果你正在建構一個代理式系統，這就是可遷移的教訓。不是具體的技術——你可能不需要基於池的渲染、KAIROS 模式或八種 MCP 傳輸。而是原則：定義你的邊界，在那裡吸收複雜性，並保持邊界之間的一切乾淨。邊界是工程困難之所在。內部是工程愉悅之所在。為愉悅的內部而設計，將你的複雜性預算投入在邊緣。

原始碼是開放的。螃蟹爪中握著地圖。去讀它吧。
