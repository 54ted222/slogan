# 第十八章：我們學到了什麼

## 五個架構賭注

Claude Code 不是唯一的 agentic 系統。它不是第一個。但它做出了五個架構賭注，讓它在 agent 框架的版圖中與眾不同。在近兩千個文件和十七個章節之後，這些賭注值得審視。

### 賭注 1：Generator Loop 勝過 Callback

大多數 agent 框架給你一個管線：定義工具，注冊處理器，讓框架協調。開發者編寫 callback。框架決定何時呼叫它們。

Claude Code 做了相反的事。`query()` 函式是一個 async generator——開發者擁有迴圈。模型串流回應，generator yield 工具呼叫，呼叫者執行它們，附加結果，generator 迴圈。有一個函式，一個資料流，一個每次互動都流過的地方。generator 返回類型的 10 個終止狀態和 7 個繼續狀態編碼了每個可能的結果。迴圈就是系統。

這個賭注是，即使一個增長到 1,700 行的單個 generator 函式，也會比分散的 callback 圖更易於理解。在研究了原始碼之後，這個賭注得到了回報。當你想理解為什麼 session 結束時，你看一個函式。當你想添加新的終止狀態時，你在一個識別聯合中添加一個變體。類型系統強制執行詳盡處理。基於 callback 的架構會將這個邏輯分散在幾十個文件中，callback 之間的交互將是隱式的，而非在控制流中可見的。

### 賭注 2：基於文件的記憶體勝過資料庫

第十一章詳細說明了這個案例，但架構意義超越了記憶體本身。選擇使用純 Markdown 文件而非 SQLite、向量資料庫或雲端服務，是一個透明優先於能力的賭注。資料庫支援更豐富的查詢、更快的查找和交易保證。文件什麼都不提供。文件提供的是信任。

用 vim 打開 `~/.claude/projects/myapp/memory/MEMORY.md` 並看到 agent 記住了關於他們的確切內容的使用者，與必須問 agent「你記住了什麼？」並希望答案完整的使用者，有著根本不同的關係。基於文件的設計使 agent 的知識狀態外部可觀察，而非只是自我報告。這比查詢效能更重要。LLM 驅動的回調系統以檢索智能補償了存儲的簡單性——一個 Sonnet 旁查詢從清單中選擇五個相關記憶體，比嵌入相似性更精確，並且需要零基礎設施。

### 賭注 3：自描述工具勝過中央協調器

Agent 框架通常提供一個工具注冊表：你在中央配置中描述你的工具，框架將它們呈現給模型。Claude Code 的工具描述自己。每個 `Tool` 物件攜帶自己的名稱、描述、輸入 schema、提示貢獻、並發安全標誌和執行邏輯。工具系統的工作不是向模型描述工具——而是讓工具描述自己。

這個賭注在可擴展性上得到了回報。MCP 工具（第十五章）透過實現相同的介面成為一等公民。來自 MCP 伺服器的工具和內建工具對模型來說是無法區分的。系統不需要單獨的「MCP 工具適配器」層——包裝產生一個標準的 `Tool` 物件，從那時起，現有的工具管線處理它：權限檢查、並發執行、結果預算、hook 攔截。

### 賭注 4：Fork Agent 用於快取共享

第九章涵蓋了 fork 機制：一個以父代完整對話作為其 context window 開始的子 agent，共享父代的提示快取。這不是方便的最佳化——而是一個架構賭注，認為快取共享模型值得 fork 生命週期管理的複雜性。

另一個選擇——用對話摘要產生新的 agent——更簡單但昂貴。每個新的 agent 都要支付從頭開始處理其上下文的全部成本。fork 的 agent 免費獲得父代的快取前綴（輸入 token 九折），使得為小任務產生 agent 是經濟的：記憶體提取、程式碼審查、驗證過程。背景記憶體提取 agent（第十一章）在每次 query loop 輪次後運行，其成本之所以微乎其微，正是因為它共享父代的快取。沒有基於 fork 的快取共享，那個 agent 的成本將高得令人望而卻步。

### 賭注 5：Hook 勝過 Plugin

大多數可擴展性系統使用 plugin——在宿主程序中注冊能力並運行的程式碼。Claude Code 使用 hook——在生命週期時間點運行的外部程序，透過 exit code 和 stdin/stdout 上的 JSON 通訊。

這個賭注是，程序隔離值得開銷。plugin 可以使宿主崩潰。hook 使自己的程序崩潰。plugin 可以將記憶體洩漏到宿主的堆中。hook 的記憶體隨其程序消亡。plugin 需要必須版本化和維護的 API 表面。hook 需要 stdin、stdout 和 exit code——一個自 1971 年以來穩定的協定。

開銷是真實的：每次 hook 調用產生一個程序，成本是毫秒，而程序內的 callback 不會有這個成本。內部 callback 的 -70% 快速路徑（第十二章）表明系統知道這個成本很重要。但對於外部 hook——使用者腳本、團隊 linter、企業政策伺服器——隔離保證使系統更安全地擴展。企業可以部署基於 hook 的政策執行，而不必擔心格式不正確的 hook 腳本會使開發者的 session 崩潰。

## 什麼可以遷移，什麼不能

並非 Claude Code 中的每個模式都能推廣。有些是規模、資源或其他 agent 構建者可能不具備的特定約束的結果。

### 可遷移到任何 Agent 的模式

**Generator loop 模式。** 任何需要串流回應、處理工具呼叫並管理多個終止狀態的 agent，都受益於使迴圈明確而非將其隱藏在 callback 後面。識別聯合返回類型——精確編碼迴圈停止的原因——是消除整類「agent 為什麼停止？」除錯 session 的模式。

**帶 LLM 回調的基於文件的記憶體。** 具體的實現細節是 Claude Code 的，但原則——簡單存儲結合智能檢索——適用於任何需要跨 session 持久化知識的 agent。四類型分類法（使用者、反饋、專案、參考）和可衍生性測試（「這能從當前專案狀態重新衍生嗎？」）是可重用的設計啟發式方法。

**遠端執行的不對稱讀/寫通道。** 當讀取是高頻串流而寫入是低頻 RPC 時，分離它們是正確的，無論具體的傳輸協定如何。

**用於搜尋的位圖預過濾。** 任何搜尋大型文件索引的 agent 受益於 26 位元字母位圖作為預過濾器。每個條目 4 位元組，每個候選者一次整數比較——成本效益比引人注目。

**提示快取穩定性作為架構關注點。** 如果你的 agent 使用帶有提示快取的 API，將穩定內容放在前面，易變內容放在後面，結構化提示，不是最佳化——而是決定你成本結構的架構決策。

### 特定於 Claude Code 規模的模式

**Fork 的終端渲染器。** Claude Code fork 了 Ink 並用緊湊的 typed array、基於池的內化和格級差分重新實現了渲染管線，因為它需要在終端中以 60fps 串流。大多數 agent 渲染到網頁介面或簡單的日誌輸出。只有當終端渲染是你的主要 UI 且你在高頻率串流時，這個工程投資才有意義。

**50 多個啟動剖析檢查點。** 當你有數十萬使用者且 0.5% 採樣產生統計上顯著的數據時才有意義。對於更小的 agent，更簡單的計時系統就足夠了。

**八種 MCP 傳輸類型。** Claude Code 支援 stdio、SSE、HTTP、WebSocket、SDK、兩種 IDE 變體和 Claude.ai 代理，因為它必須與每個部署拓撲整合。大多數 agent 需要 stdio 和 HTTP。

**Hook 快照安全模型。** 在啟動時凍結 hook 配置並且從不隱式重新讀取，是針對特定威脅的防禦：惡意的 repository 程式碼在使用者接受信任對話框之後修改 hook。這在你的 agent 在帶有不受信任的 `.claude/` 配置的任意 repository 中運行時很重要。只在受信任環境中運行的 agent 可以使用更簡單的 hook 管理。

## 複雜性的代價

近兩千個文件。這換來了什麼，代價是什麼？

文件數量作為複雜性指標具有誤導性。其中很大一部分是測試基礎設施、類型定義、配置 schema 和 fork 的 Ink 渲染器。實際的行為複雜性集中在少數高密度文件中：`query.ts`（1,700 行，agent loop）、`hooks.ts`（4,900 行，生命週期攔截系統）、`REPL.tsx`（5,000 行，互動協調器），以及記憶體系統的提示構建函式。

複雜性來自三個來源，每個都有不同的特性：

**協定多樣性。** 支援五種終端鍵盤協定、八種 MCP 傳輸類型、四種遠端執行拓撲和七個配置範圍本質上是複雜的。每個額外的協定是程式碼庫的線性添加，而非指數添加——但總和是巨大的。這種複雜性在 Brooks 意義上是偶然的：它來自環境（終端碎片化、MCP 傳輸演進、遠端部署拓撲），而非被解決的問題。

**效能最佳化。** 基於池的渲染、位圖搜尋預過濾、黏性快取鎖定和推測性工具執行，每個都以可測量的效能增益換取複雜性。這種複雜性由測量證明是合理的——每個最佳化都有識別瓶頸的剖析數據在先。風險在於最佳化會積累並以使熱路徑更難修改的方式交互。

**行為調整。** 記憶體系統的提示指令、陳舊警告、驗證協定、「忽略記憶體」反模式指令——這些不是程式碼複雜性。它們是提示複雜性，承載著不同的維護負擔。當模型版本之間的行為改變時，透過 eval 精心調整的提示指令可能需要重新調整。eval 基礎設施（在整個程式碼庫中作為案例編號和 eval 分數引用）是防止回歸的防禦，但需要持續投資。

這個系統的維護負擔是重大的。閱讀程式碼庫的新工程師不僅必須理解程式碼路徑，還要理解驅動特定提示措辭的 eval 結果、驅動特定安全檢查的生產事故，以及驅動特定最佳化的效能剖析。程式碼注釋很詳盡——許多包含 eval 案例編號和前後測量——但近兩千個文件中的詳盡注釋本身就是閱讀負擔。

## agentic 系統走向何方

從 Claude Code 的模式中可以看到四個趨勢，它們指向這個領域的走向。

### MCP 作為通用協定

第十五章將 Claude Code 描述為最完整的 MCP 客戶端之一。意義不在於 Claude Code 的實現——而在於 MCP 的存在。用於工具發現和調用的標準化協定意味著為一個 agent 構建的工具可以與任何 agent 一起工作。生態系統效應是顯而易見的：為 Postgres 構建的 MCP 伺服器一旦建成，就服務於每個使用 MCP 的 agent。開發者在工具整合上的投資是可移植的。

對 agent 構建者的啟示：如果你在定義自定義工具協定，你可能在犯一個錯誤。MCP 已經足夠好，它在變得更好，標準協定的生態系統優勢隨時間複利。構建一個 MCP 客戶端，為規格做貢獻，讓協定透過社區反饋演進。

### 多 Agent 協調

Claude Code 的子 agent 系統（第八章）、任務協調（第十章）和 fork 機制（第九章）是多 agent 模式的早期實現。它們解決了特定問題——快取共享、並行探索、結構化驗證——但也揭示了根本挑戰：協調開銷。

agent 之間的每條訊息消耗 token。每次 fork 共享一個快取，但添加了一個父代最終必須協調的對話分支。Task 系統的狀態機（排隊、運行、完成、失敗、取消）是增加複雜性而不增加能力的協調機制。隨著 agent 變得更有能力，壓力將從「我們如何協調多個 agent？」轉向「我們如何讓一個 agent 有足夠的能力，使協調成為不必要的？」

目前的證據表明兩種方法都將共存。簡單任務將使用單個 agent。複雜任務將使用協調的多 agent 系統。工程挑戰是讓協調開銷足夠低，使得交叉點有利於多 agent 用於真正並行的工作，而非只是複雜的工作。

### 持久記憶體

Claude Code 的記憶體系統是持久 agent 記憶體的第一版。基於文件的設計、四類型分類法、LLM 驅動的回調、陳舊系統和長時間運行 session 的 KAIROS 模式，都是對一個將顯著演進的問題的第一代解決方案。

未來的記憶體系統可能會添加結構化檢索（當前系統檢索整個文件；未來系統可能檢索特定事實）、跨專案遷移學習（隨處適用的使用者偏好、不適用的專案慣例），以及協作記憶體（第十一章的團隊記憶體是第一步，但同步、衝突解決和存取控制是最小的）。

開放的問題是基於文件的方法是否可擴展。在每個專案 200 個記憶體的情況下，它運作良好。在每個專案 2,000 個記憶體的情況下，Sonnet 旁查詢的清單變得太大，整合變得太昂貴，索引超過其上限。隨著使用量增長，文件優於資料庫的架構賭注將面臨最嚴峻的考驗。

### 自主操作

KAIROS 模式、背景記憶體提取 agent、自動夢境整合、推測性工具執行——這些都是走向自主操作的步驟。agent 在不被要求的情況下做有用的工作：它記住了你忘記告訴它記住的事情，在你睡覺時整合自己的知識，在當前回應完成之前就開始執行下一個工具。

軌跡是清楚的。未來的 agent 將不那麼被動，更加主動。它們會注意到使用者未描述的模式，建議使用者未要求的糾正，並在不需要明確 `/remember` 命令的情況下維護自己的知識。Claude Code 的記憶體系統，帶著其背景提取安全網和其提示工程的「什麼值得保存」啟發式方法，是這個未來的原型。

約束是信任。自主操作需要使用者信任 agent 在無人看管時會做正確的事情。基於文件的記憶體、可觀察的 hook 系統、陳舊警告、權限對話框——所有這些都存在是因為信任必須贏得，而非假設。走向更自主的 agent 之路經過更透明的 agent。

## 結語

十七章。六個核心抽象。一個 generator loop 在中心，工具向外延伸，記憶體向過去延伸，hook 守衛周邊，一個渲染引擎將一切翻譯成螢幕上的字元，MCP 將其連接到程式碼庫之外的世界。

Claude Code 最深的模式不是任何單一技術。而是反復出現的決定，將複雜性推到邊界。渲染系統將複雜性推到池和差分中——在管線內部，一切都是整數比較。輸入系統將複雜性推到分詞器和按鍵綁定解析器——在處理器內部，一切都是有類型的動作。記憶體系統將複雜性推到寫入協定和回調選擇器——在對話內部，一切都是上下文。agent loop 將複雜性推到終止狀態和工具系統——在迴圈內部，只是：串流、收集、執行、附加、重複。

每個邊界吸收混沌並輸出秩序。原始位元組成為 `ParsedKey`。Markdown 文件成為回調的記憶體。MCP JSON-RPC 成為 `Tool` 物件。Hook exit code 成為權限決策。每個邊界的一側，世界是雜亂的——五種鍵盤協定、脆弱的 OAuth 伺服器、陳舊的記憶體、不受信任的 repository hook。另一側，世界是有類型的、有界的、詳盡處理的。

如果你在建構一個 agentic 系統，這是可遷移的教訓。不是具體的技術——你可能不需要基於池的渲染或 KAIROS 模式或八種 MCP 傳輸。而是原則：定義你的邊界，在那裡吸收複雜性，保持它們之間的一切乾淨。邊界是工程困難的地方。內部是工程愉快的地方。為愉快的內部設計，並將你的複雜性預算投資在邊緣。

原始碼是開放的。螃蟹爪子裡有地圖。去讀它。
